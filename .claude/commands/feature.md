Новая фича: $ARGUMENTS

## Фаза 1: Планирование и маршрутизация
Используй agent **architect**. Он сам выберет режим (LITE/NORMAL/DEEP) и определит маршрут.

**→ ПОКАЖИ ПЛАН И ЖДИ ОДОБРЕНИЯ**

## Фаза 2: Реализация (после ОК)
Вызывай ТОЛЬКО агентов из маршрута, в порядке:
1. **agent backend** (если в маршруте) — первым, это контракт
2. **agent firmware** (если в маршруте)
3. **agent miniapp** (если в маршруте)

Между агентами передавай контракт: endpoints, JSON формат, типы.

## Фаза 3: Проверка
Используй agent **reviewer**:
- Кросс-слойная согласованность
- Edge cases
- Отчёт: 🔴 → 🟡 → 🟢

## Фаза 4: Обновление документации
После реализации проверь — изменилась ли архитектура:
- Новые endpoints или изменённые API пути?
- Новые таблицы/колонки в БД?
- Новые модули/файлы?
- Изменённый поток данных между слоями?
- Новые константы в config.h?

Если ДА хотя бы на один пункт:
1. Обнови **ARCHITECTURE.md** — дерево файлов, диаграммы, таблицы модулей, API endpoints
2. Обнови **CLAUDE.md** — если добавились новые правила, ограничения, паттерны
3. Обнови **BACKEND.md** — если изменились API endpoints

Если ничего архитектурно не поменялось — пропусти этот шаг.

## Правила
- НЕ пропускай планирование
- НЕ вызывай лишних агентов
- reviewer нужен ВСЕГДА
- обновление документации — ВСЕГДА проверяй, даже если кажется что ничего не поменялось
